용어 정리: 사용자가 언급한 “CHIFOS”는 공개 산업 표준 명칭 기준으로 CFIHOS(Capital Facilities Information Handover Specification)로 해석해 문서화했다. 사내에서 별도 CHIFOS 명칭을 사용한다면, 표준 CFIHOS와 사내 명칭의 매핑표를 별도 부록으로 추가해야 한다.

1. Executive Summary

본 제품은 정유·석유화학 공장의 설비, 자산, 문서, 구매, 사고, 정비, 기술 검토 지식을 하나의 Graph 기반 운영 지식체계로 연결하고, 사용자의 업무 질의를 메인 Router가 해석해 적절한 하위 Agent 또는 모듈로 분기하는 통합 Master Agent 플랫폼이다.

설계의 핵심은 두 가지다. 첫째, 자산과 문서의 기준 모델은 CFIHOS를 기준으로 잡고, 고장·정비·신뢰성 이벤트는 ISO 14224 개념으로 보강한다. 둘째, Graph DB는 관계 탐색과 맥락 확장에, pgvector는 의미 기반 문서 검색에, PostgreSQL Full Text Search는 키워드 검색에 사용해 Graph + Vector + Lexical Hybrid RAG 구조를 만든다.

권고 방향: “LLM 챗봇”으로 시작하지 말고, Asset-first 업무 플랫폼으로 시작해야 한다. 즉, 모든 답변은 설비/태그/문서 revision/근거/승인상태와 연결되어야 하며, 자유 답변이 아니라 업무 액션으로 이어지는 구조화된 결과물을 반환해야 한다.

2. 배경과 문제 정의

2.1 현재 문제

  • 사고보고서, ESS, API, 내부 검토 문서, RFQ, 설비 기준 정보가 문서 저장소와 업무 시스템에 흩어져 있다.
  • 설비 Tag, 물리 Equipment, Model Part, Vendor, PO, Document Revision 간 관계가 명시적으로 연결되어 있지 않다.
  • 기술 검토와 Trouble Shooting 과정에서 과거 유사 사례와 최신 revision 기준을 찾는 데 시간이 많이 든다.
  • 구매/RFQ 검토 시 요구사항, datasheet, vendor deviation, 사내 기준 간 추적성이 약하다.
  • LLM을 도입하더라도 근거 문서와 자산 맥락이 약하면 실무 승인에 쓰기 어렵다.

2.2 제품 기회

정유·석유화학 도메인은 설비 중심의 반복 업무가 많고, 문서와 이력의 관계가 의사결정 품질을 좌우한다. 따라서 설비/자산 Graph를 중심축으로 두면 구매, 기술 검토, Trouble Shooting, 자재 관리 모듈을 같은 DB와 같은 Retrieval 계층 위에서 확장할 수 있다.

3. 목표, 범위, 성공지표

3.1 제품 목표

  1. 설비·자산·문서·구매·정비·사고 데이터를 Graph 기반으로 연결한다.
  2. 메인 Router가 업무 의도를 해석해 구매, Trouble Shooting, 기술 검토, 자재 관리 모듈로 라우팅한다.
  3. RAG 답변은 항상 관련 자산, 근거 문서, revision, 누락 정보, 위험도, 후속 액션을 포함한다.
  4. 신규 모듈이 동일 Entity/Relation DB와 Knowledge Query Service를 재사용하도록 한다.
  5. 사내 Azure ML 환경에서 보안, 감사, 평가, 운영 모니터링이 가능한 구조로 구현한다.

5. 제품 구조와 모듈

제품은 하나의 Master Router와 여러 하위 Agent/업무 모듈로 구성한다. 하위 모듈은 독립적으로 발전하되, 공통 Entity/Relation Graph DB, 공통 RAG, 공통 감사 로그, 공통 승인 워크플로를 공유한다.

Master Router

사용자 질의에서 intent, asset scope, document scope, 권한, 업무 위험도를 추출하고 적절한 모듈로 라우팅한다.

Knowledge Query Service

Graph 탐색, vector 검색, keyword 검색, reranking, evidence validation을 담당하는 공통 검색 계층이다.

업무 Agent Modules

Purchase, Trouble Shooting, Technical Review, Material Management 등 업무별 정책과 출력 스키마를 가진다.

Governance Layer

Human approval, audit logging, prompt/model version, data lineage, policy check를 담당한다.

7. Entity/Relation 모델

7.1 모델링 원칙

CFIHOS를 자산/문서/구매/인수인계의 기준 모델로 두고, ISO 14224를 고장/정비/신뢰성 이벤트 모델로 보강한다. 특히 Tag와 Equipment를 분리해야 한다. Tag는 기능적 위치/요구사항에 가깝고, Equipment는 실제 물리 장비와 life history의 anchor다.

핵심 설계 결정: “같은 Tag에 다른 Equipment가 교체 장착되는 경우”와 “같은 Equipment가 이동/교체/정비되는 이력”을 동시에 추적하기 위해 Tag와 Equipment는 별도 Entity로 관리한다.

8. Graph RAG 파이프라인

8.1 인제스션 파이프라인

  1. 원본 수집: 사고보고서, ESS, API 문서, 내부 검토 문서, RFQ, 설비 기준 정보 수집.
  2. 문서 분류: document_type, discipline, plant_code, revision, confidentiality, owner 부여.
  3. 구조 파싱: heading, clause, table, datasheet row, API endpoint, action item, incident timeline 추출.
  4. Entity Linking: Tag, Equipment, ModelPart, Company, PO, DocumentRevision과 연결 후보 생성.
  5. Chunking: clause/section/row 단위로 chunk를 만들고 metadata를 부여.
  6. Embedding: Azure ML embedding endpoint를 통해 chunk embedding 생성 후 pgvector에 저장.
  7. Graph 반영: 검수된 entity/relation을 AGE graph에 upsert.
  8. 품질 점검: orphan node, stale revision, duplicate tag, missing metadata를 리포트.

8.3 Retrieval 전략

질의 정규화의도, 자산, 문서 범위, 언어 추출
Lexical Search문서번호, Tag, clause, keyword
Dense Searchpgvector semantic similarity
Graph ExpansionTag/Equipment/Document neighborhood
Rerank + Validate근거 충분성, 최신 revision, 충돌 확인
final_score = 0.40 * dense_score
            + 0.25 * lexical_score
            + 0.20 * graph_proximity_score
            + 0.15 * metadata_boost

Trouble Shooting은 graph_proximity와 failure history 가중치를 높이고, 구매/기술 검토는 document_type, revision, requirement mapping 가중치를 높인다.

8.4 답변 검증 규칙

  • 근거 없는 판단 금지. evidence가 부족하면 verdict=insufficient로 반환한다.
  • 최신 revision과 구 revision이 동시에 검색되면 최신 revision 우선, 단 과거 사고 이력은 이력으로 표시한다.
  • 안전/법규/품질 영향이 있는 경우 human approval을 필수로 둔다.
  • 서로 충돌하는 문서가 검색되면 conflict 목록과 확인 필요 부서를 표시한다.
  • 답변에는 반드시 관련 자산, 근거 문서, 누락 정보, 후속 액션을 포함한다.

9. Master Router 설계

9.1 Router의 역할

Master Router는 단순 intent classifier가 아니라 정책 기반 업무 오케스트레이터다. 사용자 질의를 Task Envelope로 정규화하고, 권한 및 자산 범위를 확인한 뒤, 모듈별 Agent에 실행을 위임한다.

기능설명우선순위
Intent classification구매, Trouble Shooting, 기술 검토, 자재 관리, 일반 검색 등으로 분류P0
Asset scope resolver질의 속 Tag/Equipment/Plant/Unit을 canonical entity로 매핑P0
Policy guard안전/위험도/승인 필요 여부 판단P0
Module dispatch하위 Agent 실행 및 결과 취합P0
Clarification handling필수 정보 누락 시 질문 생성P1
Multi-agent orchestration기술 검토 → 자재 대체품 확인 → 구매 검토처럼 복합 workflow 수행P1
Learning feedback승인/반려 이력을 routing 및 retrieval 평가에 반영P2

15. 리스크와 의사결정

데이터 품질 리스크

Tag/Equipment/Document revision 매핑이 부정확하면 RAG 품질이 무너진다.

대응: staging → 검수 → canonical 반영 프로세스와 orphan/duplicate 리포트 운영.

환각/근거 부족 리스크

LLM이 근거 없이 판단하면 실무 적용이 어렵다.

대응: evidence mandatory, insufficient verdict, human approval 기본값.

AGE 운영 리스크

Apache AGE는 PostgreSQL extension이므로 Azure 환경에서 버전/지원 범위 확인이 필요하다.

대응: 초기 PoC에서 버전 고정, migration test, fallback relational relation table 준비.

문서 파싱 리스크

PDF 표, 스캔 문서, 구형 양식은 파싱 품질이 낮을 수 있다.

대응: 문서군별 parser strategy와 수동 검수 UI를 MVP에 포함.

권한/보안 리스크

공급사 문서, 사고보고서, 프로젝트 문서에는 접근 제한이 필요하다.

대응: metadata 기반 retrieval filter, RLS, audit trail, Entra ID 권한 연계.

사용자 신뢰 리스크

초기 답변 품질이 낮으면 현업이 사용하지 않는다.

대응: Asset 360과 Evidence-first UI로 투명성을 우선 제공.

16. 부록

16.1 MVP Definition of Done

  • 선정 Plant/Unit 기준 Tag-Equipment-Document graph가 구축되어 Asset 360 화면에서 조회된다.
  • 최소 3개 문서군(ESS, 사고보고서, 내부 검토 문서 또는 RFQ)이 인제스션되고 hybrid retrieval이 가능하다.
  • Router가 4개 이상 intent를 분류하고 Task Envelope를 생성한다.
  • Technical Review 또는 Trouble Shooting 중 최소 1개 모듈이 end-to-end로 동작한다.
  • 모든 답변이 evidence, related_assets, missing_information, recommended_actions를 포함한다.
  • 승인/반려 로그가 저장되고, 반려 사유가 평가셋 개선에 사용 가능하다.
  • 기본 보안: SSO, 역할 기반 접근, 감사 로그, DB backup 정책이 적용된다.

16.2 Prompt Template 기본형

당신은 정유·석유화학 설비 검토 Agent다.
반드시 한국어로 답한다.
근거가 없는 판단은 금지한다.
우선순위는 안전 > 규정 준수 > 운전 연속성 > 비용이다.
입력된 asset scope와 document revision 범위를 벗어나면 누락 정보를 먼저 표시한다.
출력은 지정된 JSON schema를 따른다.
